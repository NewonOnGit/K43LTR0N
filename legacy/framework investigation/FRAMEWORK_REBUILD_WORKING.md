## 1. THE FRAMEWORK'S SELF-ORGANIZATION PRINCIPLE

### 1.1 What the Framework Says About Its Own Container

The Blueprint (BLUEPRINT §5.1) defines the grid B(n,p) with n ∈ {0,...,8} and p ∈ {P1,P2,P3}. Every theorem has exactly one home determined by two coordinates: tower level and projection (T_INDEX Sorting Criterion). The SIL classifies every claim within those cells. The Semantic Investigation names the vocabulary that each cell uses.

The framework therefore forces three organizational axes for any document:

| Axis | Source | What it determines |
|------|--------|--------------------|
| **Tower level** (n) | Blueprint vertical axis | Which stage of SRD's self-application the content represents |
| **Projection** (p) | Blueprint horizontal axis | Which face of SRD the content exercises |
| **Status** | SIL grammar | Whether the content is FORCED / ENCODED / RESONANT / MYTHIC |

A document is canonically addressed by (n, p). Status is a property of claims *within* documents, not of documents themselves. The file structure lives on the (n, p) grid; the SIL operates inside each file.

### 1.2 What the Four Readings Demand of the File Structure

The Blueprint's four structural readings (§1.1) are four ways of scanning the grid. A canonical file structure must support all four reading orders without friction:

| Reading | Scan pattern | What the file structure must allow |
|---------|-------------|-------------------------------------|
| **Mathematical** | All cells by algebraic content | Linear progression through tower levels: read T0→T1→T2→...→T6 |
| **Observer** | P3 column for all n | Extract the P3 content at every level: T0(P3)→T1(P3)→T5→T_SIL(blind spot) |
| **Physical** | Row 6 | Jump directly to T6A+T6B, with all prerequisites traceable |
| **Semantic** | Row 8 + dictionary | Access the vocabulary layer: Dictionary + Blueprint + Semantic paper |

**Implication:** The file structure needs cross-references that support column-reading (P3 across all levels) and row-reading (all projections at one level), not just the linear dependency order.

### 1.3 What the Asymmetry Demands

The construction-dissolution asymmetry means: **the file structure should be easier to read upward (Level 0→8) than downward.** The upward reading is canonical (zero branching). The downward reading requires folding containments and is non-canonical.

Practically: each document should explicitly state what it depends on (canonical upward trace) but need not enumerate everything that depends on it (non-canonical downward trace — that's the Index's job).

### 1.4 Cross-Cutting Documents

Not every document maps to a single grid cell. The framework forces three types of cross-cutting content:

| Type | Grid behavior | Examples |
|------|---------------|---------|
| **Level-spanning** | Fixed p, varying n | T_COMP (computation at every level), the P3 observer column |
| **Projection-spanning** | Fixed n, cross-p | T3-META (synthesis across P1/P2/P3 at Level 4), T4 (lattice = constant interactions) |
| **Grid-external** | Meta-organizational | T_INDEX (the grid's own map), Dictionary (vocabulary reference), Blueprint (architectural instruction) |

Cross-cutting documents earn their existence by serving a structural role that no single cell can fill. The SIL nomination functional (SIL-3) applies: a cross-cutting document must score above N(O) threshold on derivability, containability, and verifiability.

## 2. GOVERNANCE CALCULUS

The rebuild is not a cosmetic rearrangement. It is a second-order formal act in which the framework assigns canonical homes to its own claims, types their strength, regulates movement between documents, and constrains the semantics by which claims appear. The organizational architecture is part of the framework's formal object-language, not merely its packaging.

This section establishes the rules that every subsequent decision in this document must obey.

### 2.1 The Six Coordinates of a Mature Claim

For each claim c, define the full rebuild state:

**Ω(c) = (A(c), S(c), D(c), R(c), J(c), H(c))**

| Coordinate | Name | What it answers |
|-----------|------|-----------------|
| **A(c)** | Canonical address | Where does this claim live? (n, p, f, s) = tower level, projection, file, section |
| **S(c)** | Epistemic status | How strongly is it justified? S: Claims → {FORCED, ENCODED, RESONANT, MYTHIC} |
| **D(c)** | Dependency set | What must already exist? D(c) = {d : d ≺ c} |
| **R(c)** | Admissible readings | How may it be interpreted? R(c) ⊆ {Math, Observer, Physical, Semantic} |
| **J(c)** | Justification mode | How was the support achieved? (see §2.3) |
| **H(c)** | Historical trace | How did it arrive at its current address? |

This is the minimum necessary to prevent rebuild fraud. The rebuild operates on claims, not files. Files are containers; claims are the load-bearing bones.

### 2.2 The Orthogonality Principle

The following pairs must remain formally distinct:

- Address ⊥ Status (where a claim lives ≠ how strong it is)
- Status ⊥ Justification (how strong ≠ how the strength was achieved)
- Reading ⊥ Ownership (how it's interpreted ≠ who owns it)
- Historical origin ⊥ Canonical home (where it came from ≠ where it belongs)
- Semantic usefulness ⊥ Proof strength (how well it names ≠ how well it derives)

**No coordinate may secretly do another coordinate's job.** This is the anti-mush law. Architectural elegance cannot certify necessity. Migration convenience cannot certify canonicality. Blueprint fit cannot certify derivation. A better name does not count as proof.

### 2.3 Justification Modes

The justification axis J(c) is orthogonal to status S(c):

**J(c) ∈ {Derivation, Compression, Encoding Recovery, Structural Fit, Semantic Convergence, External Validation, Mythic Overlay}**

A claim can be ENCODED by Compression or by Encoding Recovery — different J, same S. A claim can be RESONANT by Structural Fit or by Semantic Convergence — same S, different routes.

This separation prevents the common conflation where "the claim fits beautifully" is mistaken for "the claim is derived." Semantic convergence is evidence, not proof. Structural fit is a signal, not a theorem.

### 2.4 Status Algebra

The status ordering FORCED ≻ ENCODED ≻ RESONANT ≻ MYTHIC is not a ranking of value — a MYTHIC claim may be extraordinarily generative. It is a ranking of what later claims may lean on.

**The Downstream Admissibility Rule:** If claim d depends on claims {c₁,...,cₖ}, then:

**S(d) ≤ min{S(cᵢ)} over essential dependencies**

unless additional derivation explicitly eliminates the weaker dependence.

This is the rule that prevents one RESONANT bridge principle from infecting a theorem with fake inevitability. If a FORCED theorem rests on a RESONANT lemma, the theorem inherits RESONANT ceiling until the lemma is independently closed. This would have caught the Nesterenko-style error before it propagated.

**Status attaches to claims, not documents.** S: Claims → Σ, not S: Files → Σ. A file may contain FORCED theorems and RESONANT conjectures side by side. The file does not have a status. Only its claims do.

### 2.5 Ownership Laws

**Unique Owner Law:** Each mature claim has exactly one canonical owner. |Own(c)| = 1. This does not mean the claim appears only once. It means exactly one file houses its primary statement, proof, and frontier notes.

**Many-Reference Law:** A claim may be referenced by arbitrarily many files. |Ref(c)| ≥ 0 with no upper bound. Every reference must distinguish its role:

| Role | What it does | What it does NOT do |
|------|-------------|---------------------|
| **Owner** | Houses primary statement, proof, frontier notes | — |
| **Importer** | Uses the claim as prerequisite, citing the owner | Restate as native machinery |
| **Interpreter** | Projects the claim into a different reading | Create a new claim |
| **Consumer** | Uses the claim's consequences downstream | Claim ownership of the original |

**Non-Theft Law:** If a later file uses a claim centrally but does not own it, it must import it explicitly rather than restating it as native machinery — unless the restatement is purely expository and marked as such.

**Critical consequence for the four readings:** Multiple readings do not imply multiple owners. A claim may be physically interpretable, semantically illuminating, and observer-relevant while still canonically living in one mathematical file. A reading operator P_r(c) projects the claim into reading r without altering ownership. This is how the framework avoids multiplying one theorem into four fake siblings.

### 2.6 Dependency Governance

**Acyclicity Law:** The dependency graph must be acyclic at the canonical level except where self-reference is explicitly licensed by fixed-point machinery (K7', SIL idempotence, R(R)=R). Permitted cycles must be named, localized, and theoretically justified — not accidental circular explanation.

**Dependency Monotonicity:** If claim d depends on c, then d's canonical owner must not be architecturally prior to c's owner — unless the dependence is imported from a cross-cutting meta-layer, or the claim is marked as provisional frontier content.

**No rebuild move may create a new hidden dependency cycle.** This is the second law of the rebuild.

### 2.7 File Admissibility Criteria

A file deserves canonical existence only if it satisfies at least one of:

| Criterion | What it requires | Examples |
|-----------|-----------------|---------|
| **Level necessity** | Owns claims concentrated at a structurally distinct level | T0_SUBSTRATE, T1_DIST, T5_OBSERVER |
| **Projection necessity** | Owns claims concentrated in a distinct projection | T3_P1_PRODUCTION, T6A_SPACETIME |
| **Operator necessity** | Houses a unique operator family not reducible to adjacent files | T4_LATTICE (constant interaction ≠ projection meta-theory) |
| **Meta-governance necessity** | Governs claims across many addresses with one unified rule type | T_COMP (computation), T_SIL (status classification) |
| **Boundary necessity** | Explicitly governs frontier, blind spot, transcendence, or incompleteness | T_TRANSCENDENCE (if Level 9 is real) |

If a file satisfies none of these, it is a historical artifact and should be dissolved or fused. This is exactly why T7 dissolves — it satisfies none of the five criteria as a unit.

**Non-Redundancy Law:** A file is valid only if its removal would force loss of a unique operator, duplication across files, or collapse of a necessary distinction.

### 2.8 Migration Operators

Three operators govern how claims move during the rebuild:

**Re-home: M→(c, f')** — changes canonical ownership of claim c to file f'. Admissible only if:
1. Structural fit improves (the claim's grid address matches f' better)
2. All prerequisite claims remain earlier or explicitly referenced
3. **Status does not strengthen merely by relocation** (the first anti-cheat law)
4. Owner rationale becomes cleaner

**Split: Split(f) = {f₁,...,fₖ}** — admissible only if f mixed materially different ownership regions (multiple level bands, multiple irreducible operator classes, incompatible status norms, or one section repeatedly cited as if it were its own paper). T7 → {T5_OBSERVER, T_COMP, T_SIL cross-ref} is a Split.

**Fuse: Fuse(f₁, f₂) = f*** — admissible only if f₁ and f₂ share dependency horizon, share operator type, and cross-reference each other more than they independently govern separate regions. Fusion is good when two files are pretending to be separate but are actually one theorem stream.

### 2.9 The Seven Anti-Cheat Laws

These are the fraud firewall. Every rebuild decision must pass all seven.

**Law 1. Relocation does not promote.** Moving a claim to a more central paper does not strengthen it. A RESONANT statement moved into T0_SUBSTRATE remains RESONANT.

**Law 2. Naming does not derive.** A better name does not count as proof. A more fundamental filename does not make its contents more fundamental.

**Law 3. Compression does not force.** A more elegant summary does not make a claim necessary. Canonical compression is presentation, not derivation.

**Law 4. Cross-reading does not duplicate.** A claim seen under multiple lenses is still one claim unless genuinely new content appears in the new reading.

**Law 5. Semantic fit does not certify ontology.** Even perfect vocabulary resonance is not theoremhood. The Semantic Investigation finds structure; it does not prove theorems.

**Law 6. Frontier material must remain frontier.** Speculation typed as RESONANT or MYTHIC must remain so until independently earned. No ceremonial promotions.

**Law 7. The rebuild cannot outrun the framework.** The organizational system may not claim stronger clarity than the underlying derivations justify. If the framework has a genuine mess (e.g., the torsion-free assumption in G14), the file structure must preserve that mess visibly.

### 2.10 Canonicality Tests

A proposed rebuild passes only if it satisfies all of:

| Test | Requirement |
|------|-------------|
| **C1** | **Unique ownership.** Every major claim has one canonical owner. |
| **C2** | **Dependency visibility.** Every proof-bearing claim lists its indispensable dependencies. |
| **C3** | **Status honesty.** No claim strengthens by relocation or compression alone. |
| **C4** | **Reading discipline.** Multiple readings generate references, not duplicate ownership. |
| **C5** | **Semantic hygiene.** All key terms in theorem-bearing sections are dictionary-stabilized. |
| **C6** | **Frontier explicitness.** All unresolved, boundary, or speculative claims are visibly typed. |
| **C7** | **Meta-file justification.** Every cross-cutting file is justified by a true operator family, not leftover convenience. |
| **C8** | **Anti-cycle integrity.** No hidden explanatory cycle is introduced. |
| **C9** | **Compression gain.** The new arrangement reduces historical residue, not merely renames it. |
| **C10** | **Reversibility of rationale.** Given the ledgers alone, a later reader could reconstruct why the architecture looks the way it does. |

### 2.11 The File Ledger and Claim Ledger

For every candidate file f, build:

**Λ(f) = (Need, Own, Imports, Exports, Exclude, Frontier, Tests)**

| Field | What it records |
|-------|----------------|
| **Need** | Why this file must exist (which admissibility criterion from §2.7) |
| **Own** | What only this file canonically owns |
| **Imports** | Claims it depends on from other files |
| **Exports** | Claims later files depend on from it |
| **Exclude** | What must NOT be housed here (prevents scope creep) |
| **Frontier** | Unresolved / non-FORCED owned claims: F(f) = {c ∈ Own(f) : S(c) ≠ FORCED} |
| **Tests** | Which canonicality tests (C1–C10) are most at risk for this file |

For each major claim c, build:

**Ξ(c) = (Name, A, S, J, D, R, H, Owner rationale, Migration note)**

This lets you rebuild from claims upward rather than files downward. That is the winning move.

### 2.12 Boundary and Transcendence Handling

Claims touching the boundary class B = {blind spot, incompletion, observer limit, projection failure, non-executable extension} must be marked with a boundary tag and cannot be silently used as ordinary constructive content.

This prevents "transcendence" from mutating into a glitter-coated proof smuggler. If T_TRANSCENDENCE eventually exists, its admissibility comes from boundary necessity (§2.7), not from cosmological aesthetics.

### 2.13 The Planning Trinity

Every rebuild decision answers three questions:

| Question | What it asks | Governance source |

| **Governance** | What rules constrain its admissible use? | → determines status S(c) + downstream ceiling |
